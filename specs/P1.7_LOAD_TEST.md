# [P1.7] Load Test — the P1 exit gate

> Grounded in `brand-digital-twin` @ `fb03ddd`. `tests/e2e/claim_concurrency_test.ts`
> already proves the atomic claim under concurrent workers. P1.7 widens that single
> test into a load run that *proves* P1.1 + P1.2 hold at target scale — the gate
> that closes Phase 1.

## Build
1. **Harness** (`tests/load/` or `scripts/loadtest.ts`) driving the running server
   over HTTP — reuse the concurrency test's signup→verify→login bootstrap to mint
   N tenants.
2. **Scenarios:**

   | Scenario | Load | Asserts |
   |----------|------|---------|
   | Job-claim contention | ≥2 workers, M due jobs | each job claimed exactly once (extends existing test) |
   | Sweep + healing | N concurrent tenants on `/sweep` + `/recommendations` | p95 latency within budget; no errors |
   | SSE fan-out | N concurrent EventSource connections | all receive pushed events; no dropped streams |

3. **Instrumentation read:** during the run, pull P1.2 metrics (`/metrics` or sink)
   — job lag, adapter error rate, POAS-calc p95 — and assert within budget.
   (Soft dependency on P1.2b; degrade to log-scrape if not yet landed.)
4. **Budgets** in config so the gate is objective, not eyeballed.

## Checklist
- [ ] Load harness reusing the auth bootstrap; N-tenant minting
- [ ] 3 scenarios above with pass/fail budgets
- [ ] Reads P1.2 instrumentation mid-run
- [ ] Runs in CI nightly (not per-PR — too heavy)

## Done when
Two workers show zero double-claims and latency + error rate stay within budget at
target load — the `PROD_READINESS_PLAN.md` P1 exit-gate lines.
