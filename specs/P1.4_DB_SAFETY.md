# [P1.4] DB Safety — Versioned Migrations + Backup/Restore Drill

> Grounded in upstream `brand-digital-twin` @ `fb03ddd`. Today the database is a
> single canonical `schema.sql` (Supabase/Postgres; some BigQuery-flavored DDL +
> the `claim_next_pending_job` plpgsql function at `schema.sql:346`). There is no
> versioning, no rollback path, and no proven restore. This is the
> highest-consequence open gap — data loss has no undo.

---

## P1.4a — Versioned, forward-only migrations

### Problem
`schema.sql` is applied wholesale. There's no record of *what version* a database
is at, no ordered application, and no rollback. Two engineers adding columns race;
prod and staging drift silently.

### Build
1. **`migrations/` directory**, numbered forward-only files:
   ```
   migrations/0001_init.sql            -- the current schema.sql, frozen as baseline
   migrations/0002_<change>.sql
   migrations/0003_<change>.sql
   ```
   Each file is idempotent where practical (`CREATE TABLE IF NOT EXISTS`, guarded
   `ALTER`). Optionally pair `NNNN_x.up.sql` / `NNNN_x.down.sql` for rollback.

2. **Version ledger table** (lives in the baseline migration):
   ```sql
   CREATE TABLE IF NOT EXISTS brand_twin.schema_migrations(
     version     INTEGER PRIMARY KEY,
     name        TEXT NOT NULL,
     applied_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
     checksum    TEXT NOT NULL          -- sha256 of the file; detects edited-after-apply
   );
   ```

3. **Runner** — a `SupabaseClient.runMigrations()` (or a `scripts/migrate.ts`):
   - Read `migrations/*.sql` sorted by number.
   - Compare against `schema_migrations`; apply only the unapplied, in order,
     each in its own transaction; record version + checksum on success.
   - **Checksum guard:** if an already-applied file's checksum changed, abort with
     a clear error (never silently re-run an edited migration).
   - `--dry-run` lists the pending set without applying.

4. **Mock parity:** the mock DB path (`GlobalMockDb` / `useSharedMockDb`) treats
   migrations as a no-op but `runMigrations()` must succeed (return applied=[]),
   so tests and CI don't fork behavior.

### Build checklist
- [ ] `migrations/0001_init.sql` = current `schema.sql` frozen as baseline
- [ ] `schema_migrations` ledger table
- [ ] `runMigrations()` runner: ordered, transactional, checksum-guarded, `--dry-run`
- [ ] CI step runs `migrate --dry-run` against a scratch DB on every PR
- [ ] Tests: fresh DB → all apply; re-run → zero re-applied; edited-applied file → abort

### Done when
- A fresh DB reaches current schema via the runner; a second run is a no-op; an
  edited already-applied migration is rejected; prod/staging report the same version.

## P1.4b — Automated backups + tested restore drill

### Problem
A backup you have never restored is not a backup. No backup cadence or restore
proof exists today.

### Build
1. **Automated backups:** scheduled `pg_dump` (or Supabase PITR if on their
   managed tier) — daily full + WAL/PITR where available. Encrypted at rest;
   retention policy (e.g. 30 days). Store off the primary (separate bucket).
2. **Restore drill (the part that matters):** a `scripts/restore-drill.ts` /
   CI job that, on a schedule:
   - spins a throwaway Postgres,
   - restores the latest backup into it,
   - runs `runMigrations() --dry-run` (expect: zero pending — schema matches),
   - runs a handful of read assertions (row counts on `pending_jobs`,
     `integration_state`, a tenant's orders) to prove data integrity,
   - tears down and reports pass/fail.
3. **Alarm on staleness:** if the newest backup is older than the cadence + grace,
   raise an alert (ties to P1.2 alert rules).

### Build checklist
- [ ] Scheduled encrypted backups + retention + off-primary storage
- [ ] `restore-drill` job: restore → schema-match → integrity assertions → teardown
- [ ] Backup-staleness alert wired to P1.2
- [ ] Runbook entry: manual restore steps + RTO/RPO targets

### Done when
- The restore drill is green on a throwaway DB (the PROD_READINESS_PLAN P1 exit
  gate line), and backup staleness pages someone.

## Sequencing
P1.4a (migrations) before P1.4b (the drill asserts schema-match via the runner).
Both land before P1.7 load test — you do not load-test a DB you cannot restore.
